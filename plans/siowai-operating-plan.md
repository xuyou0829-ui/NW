# SiowAI 长期运营方案

## 目标

把 SiowAI 从一个会聊天、会帮忙的助手，逐步变成一个可控、可审计、可扩展的个人技术代理，能安全地协助处理：

- 邮件起草与代发
- 密码与授权码使用
- 服务器运维
- 交换机/网络设备管理
- API 凭证与自动化调用

## 核心原则

1. **SiowAI 不是密码库**
   - 密码、授权码、API Key 存在专门的密钥系统里。
   - SiowAI 只在任务需要时读取对应一项。

2. **权限最小化**
   - 发邮件只拿发邮件的授权。
   - 管服务器只拿对应服务器的访问权限。
   - 不混用主密码、超级管理员权限、共享万能账号。

3. **确认后执行**
   - 起草、分析、排查可以默认做。
   - 对外发送、改配置、重启服务、改网络设备，需要明确确认。

4. **全程可回溯**
   - 每次重要动作要有记录。
   - 记录做了什么，不记录秘密本身。

5. **先小闭环，再大扩展**
   - 先跑通一个稳定场景。
   - 再逐步接入更多系统。

## 总体架构

### 1. 交互层

用户通过 WhatsApp / OpenClaw 与 SiowAI 对话，描述目标、材料、限制和是否允许执行。

### 2. 决策层

SiowAI 负责：

- 理解任务
- 拆步骤
- 生成草稿、命令、方案
- 在执行前请求关键确认
- 执行后总结结果

### 3. 密钥层

使用独立密钥管理系统，作为唯一真源。

推荐优先级：

1. Bitwarden（官方托管）
2. Vaultwarden（如果明确偏好自托管）
3. 其他企业级 Vault（后续再考虑）

密钥命名建议：

- `mail/freyrtech/primary-smtp`
- `server/prod/app-01/ssh`
- `network/core-switch/admin`
- `api/openai/main`
- `api/alicloud/ops`

### 4. 执行层

SiowAI 在 OpenClaw 主机上执行实际动作，包括：

- 读取指定 secret
- 连接服务器
- 调用 API
- 运行命令
- 修改文件
- 发送邮件

### 5. 审计层

保留：

- 执行动作时间
- 目标系统
- 使用的身份标签（不回显秘密）
- 执行结果
- 是否经过人工确认

## 权限模型

### A. 邮件

- 只使用 SMTP 授权码 / 应用密码
- 不使用邮箱主密码
- 每个邮箱单独建 secret
- 发信前保留人工确认

### B. 服务器

- 每台服务器单独身份
- 优先 SSH key，不推荐密码
- root 权限与普通运维权限分离
- 高风险操作保留人工确认

### C. 交换机/网络设备

- 只为明确设备建立单独凭证
- 变更前必须确认接入方式与回滚方案
- 先做只读查询，再做配置变更

### D. API

- 每个 API 单独 token
- 仅授予必需 scope
- 尽量支持轮换与撤销

## 推荐落地顺序

### 第 1 阶段，建立安全底座

1. 确定密钥库（推荐 Bitwarden）
2. 开启 2FA
3. 建立统一命名规则
4. 建立“禁止在聊天中发送主密码”的规则

### 第 2 阶段，先跑通一个闭环

先做：**写邮件 -> 确认 -> 发邮件**

原因：

- 风险可控
- 场景清晰
- 容易验证
- 能最快证明整套机制可用

### 第 3 阶段，接服务器管理

1. 整理服务器清单
2. 明确每台服务器的访问方式
3. 逐台建立密钥与标签
4. 先从只读检查开始
5. 再逐步开放变更能力

### 第 4 阶段，接网络设备

1. 先接一台交换机
2. 先做 show / 查询类操作
3. 再考虑配置变更
4. 所有网络变更保留人工确认

### 第 5 阶段，接 API 自动化

1. 先接低风险 API
2. 按服务建立 token
3. 建日志与限权规则
4. 再逐步扩到业务 API

## 推荐执行规则

### 默认可做

- 写草稿
- 看日志
- 做分析
- 查资料
- 生成方案
- 只读检查

### 必须确认后再做

- 发邮件
- 修改配置
- 重启服务
- 改交换机配置
- 调用会产生副作用的 API
- 使用新的敏感凭证

## 对 SiowXu 当前最合适的现实建议

### 现在先做的事情

1. 选定 Bitwarden 作为长期密钥库
2. 先只建立一个邮箱 secret
3. 先把邮件闭环跑通
4. 然后再扩展到服务器与网络设备

### 暂时不要做的事情

1. 不要把所有账号一股脑塞进来
2. 不要用主密码驱动自动化
3. 不要让 SiowAI 一开始就持有全局管理员权限
4. 不要在没有审计和确认边界前直接接管网络设备

## 下一步

下一步建议直接做这三件事：

1. 定 Bitwarden 目录与命名规范
2. 定第一条邮箱 secret 的字段结构
3. 定“起草 -> 确认 -> 执行”的通用工作流
