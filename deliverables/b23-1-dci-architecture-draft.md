# B23.1 机房专线接入与互联架构草案

## 1. 项目背景与假设

### 当前理解

- 我方机房：**B23.1**
- 对端：**客户数据中心**
- 互联方式：**2 条 100G 专线**
- 入楼路径：客户数据中心 -> 运营商/机房 MMR（Meet-Me Room） -> 通过光纤/尾纤引入我方 B23.1 机房
- 目标：
  - 画清楚物理与逻辑拓扑
  - 给出接入、冗余、切换、安全和设备建议
  - 整理成一个简单 PPT 供内部沟通

### 建议采用的基本原则

1. **两条 100G 线路必须物理隔离**，不要只做逻辑双链路
2. **尽量采用 L3 互联，不建议大范围 L2 透传**
3. **边界与核心分层**，不要把专线直接打到生产核心里
4. **故障切换尽量自动化**，避免人工切换成为唯一方案
5. **安全控制前置**，但不要让全流量深度检查成为 100G 瓶颈

## 2. 推荐拓扑

### 物理层建议

- 客户数据中心出 2 条独立 100G 专线
- 两条专线进入机房运营商 MMR
- 从 MMR 到 B23.1，建议继续走 **2 条独立光纤路径**
- B23.1 内部终结在 **两台独立边界设备** 上：
  - Edge-1
  - Edge-2
- 两台边界设备再分别上联到两台核心/汇聚设备：
  - Core-1
  - Core-2

### 简化拓扑图（文本版）

```text
客户数据中心
   |                         |
100G 专线 A                100G 专线 B
   |                         |
运营商/MMR-A               运营商/MMR-B
   |                         |
光纤路径 A                 光纤路径 B
   |                         |
Edge-1 ------------------- Edge-2
  |  \                     /  |
  |   \                   /   |
  |    \                 /    |
Core-1 ------------------- Core-2
        |            |
      安全域/业务区/存储区
```

## 3. 推荐架构

### 边界接入层

建议在 B23.1 放置 **两台 100G 边界设备**，专门承接客户专线。

职责：

- 对接运营商/客户专线
- 进行路由控制
- 做链路健康探测
- 承担主备或双活切换
- 与核心/汇聚层隔离

### 核心/汇聚层

建议双核心或双汇聚部署：

- Core-1
- Core-2

职责：

- 业务网段汇聚
- 路由与策略下发
- 和安全设备/服务器区互联
- 作为边界设备下游承接点

### 安全层

建议把客户互联流量单独放进 **专线互联安全域**。

最少应做：

- VRF/VLAN 隔离
- ACL / 路由过滤
- 源/目标白名单
- 东西向访问范围限制
- 管理面单独隔离

如果需要更强安全控制，可增加：

- 防火墙分区
- IDS/IPS
- 流量镜像审计

但要注意：

**100G x 2 的全流量深度安全检查，容易成为瓶颈。**

因此推荐模式：

- 基础边界控制用路由 + ACL
- 关键业务流量再进防火墙/安全设备
- 不建议所有 200G 流量都做重型深度检测

## 4. 冗余与切换建议

### 推荐优先方案：L3 + 动态路由

优先建议：

- 两条专线都做 L3 互联
- 使用 **eBGP / iBGP**（按双方网络边界决定）
- 开启 **BFD** 做快速故障探测
- 在我方边界与核心之间也做动态路由或清晰静态路由

### 切换模式建议

#### 方案 A，双活（推荐在双方能力足够时）

- 两条链路同时承载
- 利用 ECMP / BGP 路径策略分担流量
- 任一链路故障时自动收敛

优点：

- 带宽利用率高
- 故障切换自然

缺点：

- 配置复杂度更高
- 对路由设计要求更高

#### 方案 B，主备（更稳、更容易落地）

- 专线 A 主用
- 专线 B 备份
- BGP Local Preference / AS Path / MED 等方式控制主备

优点：

- 简单稳妥
- 容易运维

缺点：

- 一条链路平时闲置

### 不推荐的大方向

- 大范围二层透传
- 跨机房拉长二层广播域
- 依赖人工切换作为主要故障恢复手段

## 5. 设备建议

### 设备选型原则

1. 支持原生 100G 接口
2. 支持高可用双机部署
3. 支持 BGP / BFD / ACL / VRF
4. 有足够背板和缓冲
5. 管理和运维体系成熟

### 设备角色建议

#### 边界设备（承接专线）

优先考虑：

- 100G 路由器
- 或具备强三层能力的数据中心交换机

适合承接：

- BGP
- BFD
- 路由策略
- ACL
- VRF

#### 核心/汇聚设备

需要：

- 双机冗余
- 高密 100G / 40G / 25G 上联能力
- 稳定承接业务汇聚

### 品牌层面的建议

如果你们已有主流厂商体系，建议优先沿用现有平台，避免增加新的运维栈。

如果纯从能力看，100G 边界/核心常见方向会是：

- Cisco
- Arista
- Juniper
- H3C
- Huawei

最终不建议现在就锁型号，建议先锁：

- 架构
- 接口数量
- 冗余模式
- 是否需要硬件级 MACsec / 加密
- 是否需要防火墙串接

## 6. 安全建议

### 最基础必须做

1. 客户互联单独 VRF
2. 只开放必要网段
3. 管理面与业务面分离
4. 边界设备与核心设备分权管理
5. 日志与告警接入监控系统

### 可增强项

1. MACsec（如果链路和设备都支持）
2. DDoS / 异常流量检测
3. 流量采样与镜像
4. 专线业务单独防火墙区

## 7. 我更推荐的落地版本

如果这是第一期项目，我更推荐：

### 第一阶段推荐方案

- 2 条 100G 专线
- 2 台边界设备
- 2 台核心/汇聚设备
- **L3 互联 + BGP + BFD**
- **主备切换优先**
- 客户互联单独 VRF
- ACL 做基础边界控制
- 关键业务流量再进入安全设备

这样做的原因：

- 比全双活更稳
- 比二层拉通更安全
- 比人工切换更可控
- 比全流量重安全设备更现实

## 8. PPT 建议页数

建议做 6 页：

1. 项目背景与目标
2. 物理拓扑图
3. 推荐逻辑架构图
4. 安全设计与隔离建议
5. 冗余与切换方案
6. 设备建议与落地步骤

## 9. 当前需要确认的点

为了把 PPT 画得更准，后面最好补这几个信息：

1. 客户侧是 L2 交付还是 L3 交付
2. 双专线是否要求真正双活
3. 你们现网设备偏好品牌
4. 是否要求专线加密
5. 防火墙是否必须在专线路径上
6. 机房 B23.1 内是否已有双核心
